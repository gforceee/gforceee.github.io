滑动窗口详解

发表于 2017-05-01 | 分类于 nio | | 本文阅读量 次

介绍

一般来说TCP协议拥有以下特性：

  • 可靠性：确保数据确实到达目的地，如果未到达，能够发现并重传
  • 可控性：能够管路数据发送速度，使接受设备不过载

一般会通过滑动窗口确认机制来确保TCP协议能够完成这2个任务

TCP需要了解的一些小细节

我们都知道TCO连接的建立是需要三次握手的，建立连接后就可以传输数据了，发送方发送自己的数据，接受方收到数据之后给出一个确认的ACK包,这个包就是我们需要重点关注的地方，因为这个ACK包不单单只是回复而已，里面还包含了重要的信息。

  1. 期望接受到下一位字节序号，如果发送过来的字节序列不是期望的，便不会确认。比如说发送端发来1-20这组端数据，那么接收端在ACK包中就会返回21来确认1-20的数据都准确接收到了，下一次发可以从21开始发了，如果发送端发送32-42序号数据，那么接受端不会发送43的ACK，还是会发21的ACK
  2. 当前窗口的大小，这个主要是为了控制流量，比如说大小为S，已发送但字节为X，已确认的字节为Ａ，那么可以发送的大小Y=S-X+Ａ。

滑动窗口的具体例子

按照上面所说，我们可以将TCP会话任何时候的数据按下图分为四类：

  1. 已经发送并且得到了确认
  2. 已经发送但没有得到确认
  3. 未发送但是可以发送的数据
  4. 未发送也不允许发送的数据


TCP在数据传输过程是一个动态的过程，上述的这些数据也会实时的变换，来看一个具体例子吧

如图所示，当发送端接送到接收端发挥的ACK包里面接受的字节为36并且窗口大小不变时，滑动窗口便整个向后挪了5位，滑动窗口中的数据都是可以发送给接收端的。
这是正常情况下的数据传输，如果发送端中间发送了与预期不符合的数据，那么接受端便不会确认，TCP之中有一个重传的记时机制，如果长时间不确认，那么就会重新传输一遍数据。
